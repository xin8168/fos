# FOS 开发工作流配置

## SKILLS 自动化工作流定义

本文件定义FOS项目开发过程中的自动化工作流程。

---

## 工作流概述

```
任务开始 → 阅读主控文档 → 确认项目愿景 → 确认当前步骤
    ↓
阅读任务文档 → 执行Red阶段 → 执行Green阶段 → 执行Refactor阶段
    ↓
测试验证 → 测试通过? 
    ├── 是 → 绿灯状态 → 编写完成报告 → 还有下一步?
    │                                   ├── 是 → 自动启动下一步
    │                                   └── 否 → 阶段完成
    └── 否 → 修复问题 → Red阶段
```

---

## SKILL-001: 任务启动检查

### 触发条件
- 每个任务步骤开始时
- 任务中断后恢复时

### 执行步骤

1. **阅读主控文档**
   - 动作: 读取文件
   - 路径: docs/FOS-MASTER-CONTROL.md
   - 必须: 是

2. **确认项目愿景**
   - 动作: 验证确认
   - 内容:
     - 最高目标: 零风险、零兜底、零混乱地控制任何设备
     - 产品定位: 确定性执行主板（非AI推理引擎）
     - 交付标准: 全量、全功能、工业级产品
   - 必须: 是

3. **确认当前步骤**
   - 动作: 验证确认
   - 内容: 当前执行步骤编号
   - 必须: 是

4. **确认完成度**
   - 动作: 验证确认
   - 内容: 当前项目完成度百分比
   - 必须: 是

5. **阅读任务文档**
   - 动作: 读取文件
   - 路径: docs/tasks/phase-X/STEP-XXX.md
   - 必须: 是

### 验收标准
- 主控文档已阅读
- 项目愿景已确认
- 当前步骤已确认
- 完成度已确认
- 任务文档已阅读

## SKILL-002: TDD开发循环

### 触发条件
- 任务文档阅读完成后
- 测试失败需要修复时

### Phase: Red（编写失败的测试）

1. **分析需求**
   - 输入: 任务文档
   - 输出: 需求列表
   - 规则: 每个需求必须可验证

2. **设计测试用例**
   - 输入: 需求列表
   - 输出: 测试用例设计
   - 规则:
     - 每个需求至少一个测试
     - 测试必须可验证
     - 测试必须能失败

3. **编写测试代码**
   - 输入: 测试用例设计
   - 输出: 测试文件
   - 规则:
     - 使用标准测试框架
     - 测试命名清晰
     - 包含边界测试

4. **运行测试**
   - 命令: cargo test --workspace
   - 预期: 失败（功能未实现）

### Phase: Green（实现最小代码）

1. **实现最小代码**
   - 输入: 测试文件
   - 输出: 实现代码
   - 规则:
     - 只实现让测试通过的最小代码
     - 不添加额外功能
     - 不优化性能

2. **运行测试**
   - 命令: cargo test --workspace
   - 预期: 通过

3. **检查覆盖率**
   - 命令: cargo tarpaulin --workspace
   - 阈值: 90%

### Phase: Refactor（优化重构）

1. **分析代码异味**
   - 工具: cargo clippy
   - 输出: 代码问题列表

2. **重构代码**
   - 规则:
     - 保持测试通过
     - 不改变行为
     - 提高可读性
     - 消除重复

3. **验证重构**
   - 命令: cargo test --workspace
   - 命令: cargo fmt --check
   - 命令: cargo clippy --workspace -- -D warnings

## SKILL-003: 任务完成验证

### 触发条件
- TDD循环完成后
- 提交代码前

### 验证清单

| 序号 | 检查项 | 命令 | 必须 |
|-----|-------|------|-----|
| 1 | 运行所有测试 | cargo test --workspace --all-targets | 是 |
| 2 | 代码格式检查 | cargo fmt --check | 是 |
| 3 | Clippy检查 | cargo clippy --workspace -- -D warnings | 是 |
| 4 | 覆盖率检查 | cargo tarpaulin --workspace --out Xml | 是 |
| 5 | 安全审计 | cargo audit | 是 |
| 6 | 文档生成 | cargo doc --workspace --no-deps | 是 |

### 绿灯标准

```
绿灯条件：
├── [PASS] 所有测试通过
├── [PASS] 代码格式正确
├── [PASS] Clippy检查通过
├── [PASS] 覆盖率达标 (≥90%)
├── [PASS] 无安全漏洞
└── [PASS] 文档生成成功
```

## SKILL-004: 自动启动下一个任务

### 触发条件
- 当前任务绿灯状态
- 存在下一个任务

### 执行步骤

1. **编写完成报告**
   - 路径: docs/reports/STEP-XXX-report.md
   - 内容: 完成报告模板

2. **更新进度**
   - 文件: docs/FOS-MASTER-CONTROL.md
   - 更新: 当前步骤编号

3. **检查下一任务**
   - 路径: docs/tasks/phase-X/STEP-XXX.md

4. **通知用户**
   - 消息模板:
     ```
     🟢 当前任务已完成
     📋 下一个任务: STEP-XXX
     📄 任务文档: docs/tasks/phase-X/STEP-XXX.md
     请确认是否继续执行下一个任务？
     ```

5. **自动继续**
   - 条件: 用户确认继续
   - 动作: 触发SKILL-001

## SKILL-005: 上下文管理

### 触发条件
- Token使用率达到阈值时
- 长时间任务执行时

### 压缩策略

| Token使用率 | 动作 | 说明 |
|------------|------|------|
| 80% | 警告 | 准备压缩，提醒用户 |
| 85% | 压缩 | 执行压缩策略 |
| 90% | 紧急 | 立即压缩，仅保留核心 |

### 压缩规则

**必须保留**:
- 项目愿景目标
- 最高用户需求
- 当前执行步骤
- 强制约束清单
- FOS九大铁律
- 拒绝清单
- 当前任务目标

**可以压缩**:
- 已完成步骤摘要化
- 详细实现细节引用

**可以归档**:
- 详细完成报告
- 历史讨论内容

### 防止失智措施

```
防止失智检查清单：
├── [ ] 每个任务开始前重读主控文档
├── [ ] 定期确认项目愿景不变
├── [ ] 关键决策记录在任务文档
├── [ ] 使用检查清单验证执行
├── [ ] 不偏离核心目标
└── [ ] 不遗漏强制约束
```

## 使用流程

### 1. 开始新任务
```
触发 SKILL-001（任务启动检查）
  → 阅读主控文档
  → 确认项目愿景
  → 阅读任务文档
  → 执行TDD循环
```

### 2. TDD开发流程
```
触发 SKILL-002（TDD开发循环）
  → Red: 编写失败测试
  → Green: 实现最小代码
  → Refactor: 优化重构
  → 验证绿灯状态
```

### 3. 完成验证
```
触发 SKILL-003（任务完成验证）
  → 运行所有测试
  → 代码格式检查
  → Clippy静态检查
  → 覆盖率检查
  → 安全审计
  → 文档生成
```

### 4. 自动继续
```
触发 SKILL-004（自动启动下一个任务）
  → 编写完成报告
  → 更新进度
  → 检查下一任务
  → 自动启动
```

---

**注意**: 本配置定义的工作流程是强制性的，所有开发任务必须遵循此流程执行。
