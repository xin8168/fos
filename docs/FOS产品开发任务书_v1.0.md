## 一、核心需求分析

### 1.1 最高用户需求

**"零风险、零兜底、零混乱地控制任何设备"**

用户真正需要的不是"AI能控设备"，而是：
- 一个**隔离、可回滚、只执行成功过的设备操作的确定性层**
- 让设备操作**100%可预测、100%可追溯、100%可恢复**

### 1.2 用户真实痛点（深层需求）

| 表层需求 | 深层需求 | 用户不知道的真相 |
|---------|---------|-----------------|
| 用FOS控制设备执行任务 | 不想因设备操作出错造成不可逆损失 | 需要的是「隔离、可回滚的确定性层」 |
| 管理多设备/工具的SKILLS | 不想为不同设备重复写执行代码 | 需要的是「一次定义、全设备通用的标准化SKILL」 |
| 确保设备操作安全可控 | 不想让设备操作越权、跨边界 | 需要的是「沙箱隔离+自动回滚的零信任机制」 |
| 统一管控多设备/工具 | 不想记各种设备的指令格式 | 需要的是「统一明文入口+统一设备管控协议」 |

### 1.3 行为数据锚定

1. **90%的安全事故**源于「AI执行了计划外的设备操作」
2. **75%的开发量**消耗在「设备适配、操作隔离、失败回滚、权限管控」
3. **80%的复用时间**消耗在「重新调试操作步骤、排查环境差异」

### 1.4 核心价值主张

> **FOS，让每一个用户，都成为数字/物理设备的确定性掌控者——只执行成功过的操作，只做计划内的动作，零风险、零兜底、零混乱**

### 1.5 目标用户群体

| 用户群体 | 身份定位 | 核心诉求 |
|---------|---------|---------|
| 企业运维/设备管理员 | 设备操作的零风险掌控者 | 企业设备操作资产的安全管理 |
| 个人极客/专业用户 | 个人设备的自动化主权者 | 个人成功操作的复用库，一键执行零出错 |
| 工业/物联网开发者 | 工业设备的标准化执行者 | 一次定义SKILL，全产线设备通用 |

### 2.2 详细旅程节点

#### 阶段1：需求产生 → CLI输入

| 触发场景 | 用户行为 | 系统响应 | 成功标准 |
|---------|---------|---------|---------|
| 需要清理桌面文件 | 输入`fos run --事件 "清理桌面"` | 提示补充缺失字段 | 10秒内完成补全 |
| 复用历史任务 | 输入`fos reuse "清理桌面"` | 从硬记忆库调取 | 一键触发执行 |
| 查看任务状态 | 输入`fos status` | 返回明文状态 | 实时反馈 |

#### 阶段2：系统校验 → 安全执行

| 校验环节 | 校验内容 | 失败处理 | 体验保障 |
|---------|---------|---------|---------|
| 格式校验 | 6维+4要素完整 | 返回缺失字段提示 | 中文提示，无技术术语 |
| 规则校验 | 权限/红线/可执行性 | 返回拦截原因 | 明确原因+修复建议 |
| 沙箱预执行 | 环境一致性+步骤验证 | 自动回滚+返回差异 | 失败不污染真实环境 |

#### 阶段3：执行反馈 → 记忆存储

| 执行结果 | 系统行为 | 用户感知 | 后续动作 |
|---------|---------|---------|---------|
| 成功 | 写入硬记忆库 | "执行成功，已移动5个文件" | 可一键复用 |
| 失败 | 写入拦截日志 | "步骤X失败：原因XXX" | 查看日志+修复后重试 |
| 拦截 | 记录拦截原因 | "无XX权限/违反XX红线" | 申请权限或修改指令 |

### 3.2 架构设计原则

| 原则 | 说明 | 实现方式 |
|-----|------|---------|
| **主板不可变** | 原有5大模块是主板本体，不可动、不可改、不可绕过 | 硬编码锁死，版本控制 |
| **单向数据流** | 所有指令只有一条路径，必须顺序通过每个环节 | 流水线模式，无分支 |
| **物理隔离** | 核心模块与扩展模块独立部署，内存/CPU/网络隔离 | K8s命名空间 + 容器隔离 |
| **幂等执行** | 同一指令重复调用，结果一致 | 唯一执行令牌 + 状态锁定 |
| **失败即回滚** | 任意环节失败，立即终止并恢复状态 | 快照机制 + 回滚脚本 |

### 4.2 核心模块详细设计

---

#### 模块1: Gateway 协议网关

**模块定位**: FOS主板唯一对外端口，所有指令必须由此进入

**功能边界**:
```
✓ 接收外部明文指令
✓ 校验6维锚定+4要素格式
✓ 生成合规令牌
✓ 转发到校验引擎
✗ 不做规则判断
✗ 不修改指令内容
✗ 不直接访问设备
```

**核心数据结构**:

```rust
/// FOS 6维锚定结构
pub struct SixAnchor {
    pub event: String,           // 事件: 用户目标
    pub steps: Vec<String>,      // 步骤: 执行序列
    pub judgment_logic: String,  // 判断逻辑
    pub verification_standard: String, // 校验标准
    pub location: String,        // 地点: 执行环境
    pub subject: String,         // 主体: 执行者
}

/// FOS 4要素执行结构
pub struct FourElement {
    pub action: String,          // 动作
    pub target: String,          // 对象
    pub condition: String,       // 条件
    pub expected_result: String, // 预期结果
}

/// 合规令牌
pub struct ComplianceToken {
    pub token_id: String,        // 唯一ID
    pub command_id: String,      // 关联命令
    pub issued_at: DateTime<Utc>,
    pub expires_at: DateTime<Utc>, // 有效期5分钟
    pub signature: String,       // 防篡改签名
}
```

**接口设计**:

```rust
// POST /api/v1/gateway/validate
// 校验明文指令格式
pub async fn validate_format(cmd: FosCommand) -> Result<ComplianceToken>;

// GET /api/v1/gateway/health
// 健康检查
pub async fn health_check() -> Result<HealthStatus>;
```

**验收标准**:
- [ ] 格式校验响应时间 < 100ms
- [ ] 格式错误拦截率 100%
- [ ] 非明文内容拒绝率 100%
- [ ] 令牌有效期精确控制 ±1s

#### 模块2: Validator 规则校验引擎

**模块定位**: FOS主板规则裁判，所有执行必须通过校验

**功能边界**:
```
✓ 权限校验（用户角色、操作权限）
✓ 红线规则校验（危险操作拦截）
✓ 可执行性校验（设备状态、环境检查）
✓ 生成校验结果报告
✗ 不执行具体操作
✗ 不访问真实设备
✗ 不修改指令内容
```

**核心规则类型**:
