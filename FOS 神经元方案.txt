# FOS 确定性执行主板 · 扩展工具模块整合重塑方案
## 核心锚点（新增模块遵循FOS铁律）
新增CLI/SKILLS/沙箱/MCP扩展工具管理模块后，仍坚守**FOS=纯规则、无模型、零推理、只执行「明确/成功过」任务的确定性执行主板**，所有设备/工具操作均以“明文硬规则+成功事件复用+沙箱隔离”为核心，绝不执行计划外动作、绝不碰模糊指令、绝不跨边界操作。

## 一、深层痛点洞察（新增设备/工具执行维度）
### 补充行为数据锚定
AI+设备/工具执行场景的共性行为特征：
1.  企业/个人用户用AI控制电脑/设备，**90%的安全事故源于「AI执行了计划外的设备操作（删文件/改配置/控硬件）」，仅10%源于操作效率低**；
2.  开发者做设备控制类AI应用，**75%的开发量消耗在「设备适配、操作隔离、失败回滚、权限管控」，仅25%用于核心业务逻辑**；
3.  用户复用设备操作任务时，**80%的时间消耗在「重新调试操作步骤、排查环境差异、修正执行错误」，仅20%用于任务本身**。

## 六、新增扩展工具模块：终极目标、功能设计、体验优化
所有新增模块均与原有5大核心模块无缝衔接（需通过协议网关格式校验、校验引擎规则校验、执行总线调度、硬记忆库留存成功操作），全程坚守FOS核心铁律。

### 模块6：CLI 明文交互入口（FOS唯一人机交互层）
#### 模块终极目标
成为用户与FOS的「极简明文对话窗口」，仅接收符合FOS协议的中文明文指令，让用户无需学习复杂命令、无需懂技术，就能精准管控所有设备/任务。
#### 功能设计
1.  **纯中文明文指令集**：仅支持符合FOS 6维锚定+4要素执行的中文明文指令，例：
    ```bash
    # 用户CLI输入示例（极简格式，自动补全缺失字段提示）
    fos run --事件 "清理电脑桌面无用文件" --步骤 "1.列出桌面文件 2.筛选7天未修改文件 3.移动到归档文件夹" --判断逻辑 "文件大小<100MB且7天未修改" --校验标准 "归档文件夹出现对应文件，桌面无残留" --地点 "我的Windows电脑" --主体 "我"
    ```
2.  **指令自动补全/校验**：用户输入不完整时，仅提示补充缺失的明文字段（不做任何推理/生成），非标准指令直接返回明确中文原因；
3.  **成功任务一键复用**：支持通过CLI调用硬记忆库中的成功事件，例：`fos reuse "清理电脑桌面无用文件"`，自动复用历史成功的操作步骤；
4.  **执行状态实时反馈**：CLI实时输出明文执行状态（「执行步骤1：列出桌面文件→完成」「执行步骤2：筛选文件→完成」），失败则输出「步骤X失败：原因XXX」；
5.  **权限分级管控**：管理员可通过CLI定义「普通用户仅能执行、不能修改规则」「高危操作仅管理员可发起」，所有CLI操作全留痕。
#### 体验优化
- 无需记忆复杂参数/命令，仅用自然中文明文描述任务，CLI自动适配为FOS标准格式；
- 输入错误/格式不符时，仅给出「缺XX字段，请补充」的中文提示，无技术术语、无晦涩报错；
- 成功任务复用仅需输入事件名，无需重复输入步骤，一键触发执行，全程10秒内完成。

### 模块7：SKILLS 标准化执行单元（设备操作的最小原子层）
#### 模块终极目标
成为「成功设备操作的标准化复用库」，所有SKILLS均为明文定义的原子操作步骤，仅留存成功过的操作，绝不执行未验证的步骤，适配所有设备/系统。
#### 功能设计
1.  **SKILLS 纯明文定义规范**：所有SKILLS必须基于FOS 4要素执行规则，以明文JSON/YAML定义，例：
    ```yaml
    # SKILLS定义示例（清理桌面文件）
    事件ID: SKILL-20240308-001
    执行步骤:
      1: cmd /c dir C:\Users\XXX\Desktop > file_list.txt
      2: python filter_files.py file_list.txt --days 7 --size 100MB
      3: cmd /c move @filtered_files C:\Users\XXX\Archive
    判断逻辑: 步骤1返回码=0 且 步骤2筛选出文件数≥0 且 步骤3返回码=0
    校验标准: Archive文件夹存在筛选出的文件，Desktop无对应文件
    适配设备: Windows/macOS/Linux（自动适配命令）
    ```
2.  **SKILLS 仅接收硬记忆库验证**：新SKILLS必须先在沙箱执行成功并写入硬记忆库，才能加入可用库，未验证的SKILLS禁止调度；
3.  **设备自动适配**：SKILLS内置设备适配插件，同一SKILLS自动适配不同系统/设备的命令格式（如Windows的cmd vs Linux的bash），用户无感知；
4.  **SKILLS 版本锁止**：成功的SKILLS版本不可修改，仅可新增版本，防止修改导致执行失败，所有版本全留痕；
5.  **SKILLS 权限隔离**：不同主体（用户/部门）仅能调用授权的SKILLS，越权调用100%拦截。
#### 体验优化
- 新增SKILLS仅需按模板填写明文步骤，无需懂编程，设备适配由插件自动完成；
- SKILLS执行失败时，自动回滚到执行前状态，用户无需手动恢复设备/系统；
- 成功的SKILLS可一键分享/复用，跨设备执行无差异，无需重新调试。

### 模块8：沙箱隔离执行层（设备操作的安全边界）
#### 模块终极目标
成为FOS执行设备操作的「绝对安全隔离罩」，所有操作均在沙箱内执行，失败自动回滚，绝不影响真实环境，彻底杜绝设备操作风险。
#### 功能设计
1.  **全场景沙箱覆盖**：
    - 数字环境：文件操作沙箱（隔离真实文件系统）、系统命令沙箱（隔离系统调用）、网络操作沙箱（隔离真实网络）；
    - 物理设备：IoT/工业设备沙箱（模拟设备状态，验证操作无误后再下发真实指令）；
2.  **强制沙箱预执行**：所有设备操作SKILLS，必须先在沙箱内执行成功，才能下发到真实设备，沙箱执行失败则直接终止；
3.  **自动回滚机制**：沙箱/真实环境执行失败时，自动触发回滚脚本，恢复到操作前的设备/系统状态，无残留、无混乱；
4.  **沙箱与真实环境一致性校验**：执行前自动校验沙箱与真实环境的核心参数（文件路径/设备状态/系统版本），不一致则提示并终止；
5.  **沙箱日志独立留存**：沙箱内所有操作明文日志独立留存，与真实执行日志对比，便于排查差异。
#### 体验优化
- 沙箱自动启动/销毁，用户无需手动配置，仅感知「操作前先验证，绝对安全」；
- 沙箱执行失败时，返回「真实环境与沙箱差异：XXX」的明确原因，用户一键修复差异后可重新执行；
- 高危操作（删除文件/修改设备配置）在沙箱验证后，额外触发人工确认，确认后才执行真实操作。

### 模块9：MCP 设备/工具统一管控总线（FOS设备接入层）
#### 模块终极目标
成为FOS与所有数字/物理设备的「统一明文管控网关」，所有设备接入均通过MCP转换为FOS标准协议，用户无需适配不同设备的原生协议。
#### 功能设计
1.  **设备标准化接入**：所有设备（电脑/手机/IoT/工业设备）通过MCP插件接入，插件将设备原生协议（如Modbus/HTTP/MQTT）转换为FOS中文明文协议，FOS核心绝不碰设备原生协议；
2.  **设备状态明文同步**：MCP实时将设备状态（如「电脑桌面文件数」「IoT设备温度」）转换为FOS 6维锚定的中文明文数据，写入FOS状态库；
3.  **设备操作统一调度**：FOS执行总线通过MCP向所有设备下发标准化明文操作指令，MCP自动转换为设备原生指令，操作结果明文返回；
4.  **设备离线/异常处理**：设备离线/异常时，MCP自动缓存指令，设备上线后按规则执行，异常状态明文通知用户；
5.  **设备权限统一管控**：所有设备的操作权限通过FOS硬规则定义，MCP严格执行，越权操作100%拦截。
#### 体验优化
- 新增设备仅需安装对应MCP插件，无需修改FOS核心配置，5分钟内完成接入；
- 所有设备的状态/操作结果均以统一中文明文展示，用户无需懂设备原生协议；
- 设备异常时，MCP自动尝试恢复连接，用户仅收到「设备XXX离线，已缓存指令」的中文通知，无感知中断。

# FOS 主板 + 扩展模块 终极衔接方案（唯一合法链路）
## 一、衔接4条铁律（绝对不能破）
1. **所有扩展模块，都不允许直接访问原有5大核心**，只能通过「FOS明文协议网关」进出。
2. **所有指令/任务/设备操作，必须先过协议网关 → 校验引擎 → 调度总线**，一步不能少、一步不能跳。
3. **只有成功执行完的任务，才能进入硬记忆库；只有被拦/失败的，才能进拦截日志**。
4. **SKILLS/沙箱/MCP 只做“执行/转发/隔离”，绝对不做规则判断、不做格式修改**。

## 二、关键衔接控制点（确保不偏离主板核心）
### 1. 唯一入口：所有指令必须走CLI→协议网关
- **禁止**：SKILLS/MCP/沙箱直接接收外部指令；
- **强制**：哪怕是设备自动触发的任务（如定时清理文件），也必须先封装为FOS明文指令，通过CLI接口进入协议网关；
- **落地手段**：协议网关是FOS主板唯一对外端口，所有扩展模块无独立对外接口。

### 2. 唯一判断：所有规则判断必须走校验引擎
- **禁止**：沙箱/MCP/SKILLS做任何权限/规则判断；
- **强制**：哪怕是“设备是否在线”的判断，也必须先通过校验引擎的「可执行性校验」；
- **落地手段**：校验引擎输出的「规则合规令牌」是执行总线调度的唯一凭证，无令牌则调度总线拒绝工作。

### 3. 唯一执行：所有设备操作必须走执行总线→SKILLS→MCP
- **禁止**：绕过执行总线直接调用SKILLS/MCP；
- **强制**：执行总线记录每一步SKILLS/MCP的调用日志，无总线指令则SKILLS/MCP拒绝执行；
- **落地手段**：SKILLS/MCP仅接收执行总线的「明文调度指令+合规令牌」，否则拒绝响应。

### 4. 唯一归档：所有结果必须走执行总线→硬记忆库/拦截日志
- **禁止**：SKILLS/沙箱/MCP直接写入记忆库/日志；
- **强制**：只有执行总线确认“执行成功且符合校验标准”，才能触发硬记忆库写入；
- **落地手段**：硬记忆库/拦截日志仅开放给执行总线的写入接口，其他模块无写入权限。

  ```python
  # 仅接收明文，非明文直接返回400
  @app.post("/api/fos/gateway/validate")
  def validate_format(plain_text_cmd: dict):
      # 仅校验6维+4要素字段是否存在、是否为字符串
      required_fields = ["时间","地点","主体","起因","经过","结果","事件名","执行步骤","判断逻辑","校验标准"]
      for field in required_fields:
          if field not in plain_text_cmd or not isinstance(plain_text_cmd[field], str):
              return {"code":400, "msg":f"格式错误：{field}缺失或非明文"}
  ```

## 三、规则层：硬编码锁死衔接逻辑（杜绝人为/程序篡改）
### 1. API版本锁死（衔接接口不兼容）
- 原有模块对外API（网关/校验引擎/总线）：版本号硬编码（如v1.0.0），仅支持该版本，新增版本需全量测试后单独部署，不影响旧版本；
- 扩展模块调用核心API：必须携带版本号，版本不匹配直接拒绝，避免接口兼容问题导致的衔接失败。

### 2. 指令格式硬校验（无模糊空间）
- 协议网关对指令的校验逻辑：硬编码在核心模块，不支持动态配置，仅能通过版本升级修改；
- 校验规则：6维+4要素字段必须是字符串、长度≤1024字符、无特殊符号（仅中文/数字/标点），不符合则直接拦截，无“模糊通过”选项。

### 3. 执行步骤不可跳（机械执行不跑偏）
- 执行总线调度SKILLS：硬编码“顺序执行”逻辑，上一步返回“成功”才能执行下一步，返回“失败”则立即终止，无“强制跳过”开关；
- SKILLS执行步骤：步骤数、步骤内容写入硬记忆库后不可修改，仅能新增版本，避免步骤篡改导致的执行混乱。

## 四、故障兜底层：全链路失败自动处理（不慌、不卡、不脏）
### 1. 超时重试+熔断降级（避免卡壳）
| 衔接环节 | 超时阈值 | 重试次数 | 熔断降级策略 |
|----------|----------|----------|--------------|
| CLI→协议网关 | 100ms | 0（格式错误无需重试） | 直接返回格式错误提示 |
| 协议网关→校验引擎 | 200ms | 1 | 重试失败则写入拦截日志，返回规则校验超时 |
| 校验引擎→执行总线 | 200ms | 1 | 重试失败则返回系统繁忙，任务进入待执行队列 |
| 执行总线→沙箱 | 500ms | 2 | 重试失败则降级为“仅校验步骤，不执行设备操作” |
| 沙箱→SKILLS | 500ms | 2 | 重试失败则触发回滚，写入拦截日志 |
| SKILLS→MCP | 300ms | 2 | 重试失败则降级为“使用备用MCP插件” |
| MCP→设备 | 1000ms | 3 | 重试失败则标记设备离线，任务暂停并通知用户 |

### 2. 自动回滚（失败必恢复）
- **数字环境回滚**：沙箱执行前自动对设备/系统状态做快照（如文件列表、注册表、配置文件），失败则通过快照恢复；
- **物理设备回滚**：SKILLS内置“回滚步骤”（如“移动文件”的回滚步骤是“移回原路径”，“修改设备参数”的回滚步骤是“恢复原参数”），失败则自动执行回滚步骤；
- **回滚验证**：回滚后执行总线自动校验设备/系统状态是否与执行前一致，不一致则通知人工介入。

### 3. 异常隔离（故障不扩散）
- 扩展模块异常：采用“舱壁模式”，单个SKILLS/MCP插件异常，仅隔离该插件，其他插件正常运行（如“清理桌面”SKILLS崩溃，不影响“备份文件”SKILLS）；
- 设备异常：MCP自动将异常设备加入“黑名单”，暂停向该设备下发指令，直至人工解除，避免反复调用异常设备导致流水线阻塞。

## 五、监控层：全链路可观测（问题早发现、早定位）
### 1. 核心监控指标（明文采集、实时展示）
| 指标类型 | 监控维度 | 告警阈值 | 告警方式 |
|----------|----------|----------|----------|
| 延迟指标 | 每个衔接环节的端到端延迟 | 超过阈值2倍（如网关校验延迟>200ms） | 钉钉/短信+CLI明文提示 |
| 成功率指标 | 每个环节的执行成功率 | <99.9% | 钉钉/短信+日志标红 |
| 失败指标 | 失败类型（格式错误/规则拦截/沙箱失败/MCP失败） | 单类型失败率>1% | 钉钉/短信+失败原因汇总 |
| 资源指标 | 核心模块CPU/内存/磁盘，扩展模块资源使用率 | CPU>80%/内存>80%/磁盘>90% | 钉钉/短信+自动扩容 |
| 设备指标 | 设备在线率、响应率 | 在线率<95%/响应率<99% | 钉钉/短信+设备状态列表 |

### 2. 全链路明文日志（可回溯、可审计）
- 日志内容：每个衔接环节的「执行令牌+时间戳+输入数据+输出数据+执行状态+耗时」，全部中文明文；
- 日志存储：按“执行令牌”聚合，一键查询某条任务的完整链路日志，无需跨模块拼接；
- 日志分析：自动识别高频失败原因（如“沙箱失败Top3：环境不一致、步骤错误、权限不足”），输出优化建议。

### 3. 实时链路追踪（可视化排查）
- 基于OpenTelemetry实现链路追踪，每个任务的执行链路以时序图展示，鼠标悬停可查看每个环节的详细信息；
- 异常链路自动标红，点击可直接定位到失败环节，展示失败原因和建议解决方案。

## 六、测试层：全场景验证（上线前堵死漏洞）
### 1. 衔接点专项测试（每个接口必测）
| 测试类型 | 测试内容 | 验收标准 |
|----------|----------|----------|
| 功能测试 | 每个衔接环节的输入/输出是否符合预期 | 100%符合硬规则，无模糊结果 |

| 并发测试 | 10万条任务并发执行 | 成功率100%，延迟波动≤±5ms |

### 2. 混沌测试（主动造故障，验证兜底）
| 故障场景 | 注入方式 | 验收标准 |
|----------|----------|----------|
| 核心模块单节点宕机 | 手动关闭主节点 | 自动切换到备节点，任务无中断，成功率100% |
| 沙箱模块崩溃 | 手动kill沙箱进程 | 流水线自动降级，核心模块正常，任务回滚成功 |

| 硬记忆库只读 | 手动设置存储卷为只读 | 仅影响新任务写入，历史任务可正常复用，无崩溃 |

### 3. 回归测试（版本升级不翻车）
- 每次扩展模块升级，必须全量回归测试所有衔接环节，确保与核心模块的兼容性；
- 回归测试用例固化（≥1000条），覆盖所有正常/异常场景，测试通过率100%才能上线。

## 七、部署层：灰度发布+热更新（上线无风险）
### 1. 灰度发布（逐步放量，发现问题及时回滚）
- 扩展模块上线：先灰度到10%用户，观察24小时无异常后，再放量到50%，最后全量；
- 核心模块升级：先部署到测试环境，验证通过后，采用“蓝绿部署”，旧版本保留至新版本稳定运行72小时。

### 2. 热更新（不中断流水线）
- 扩展模块支持热更新：无需重启核心模块，仅替换插件容器，流水线无感知；
- 核心模块升级：采用“滚动更新”，逐个替换节点，确保集群始终有可用节点，任务无中断。

## 八、关键衔接点专项保障（最容易出问题的地方）
| 关键衔接点 | 专项保障措施 |
|------------|--------------|
| CLI→协议网关 | 1. CLI输入自动补全字段，减少格式错误；2. 输入缓存本地，断网后重连自动提交；3. 格式错误实时提示，无需等待网关返回 |
| 校验引擎→执行总线 | 1. 校验引擎输出的“合规令牌”带有效期（5分钟），避免令牌滥用；2. 总线接收令牌后先校验签名，防止令牌篡改 |
| 沙箱→SKILLS | 1. 沙箱预执行后生成“环境一致性校验码”，SKILLS执行前再次校验；2. SKILLS步骤加密传输，防止篡改 |
| MCP→设备 | 1. MCP与设备通信采用加密通道（TLS1.3），防止指令被劫持；2. 设备返回结果做签名校验，防止伪造执行结果 |

## 总结
### 稳定性核心逻辑
1. **物理隔离+集群部署**：从架构上杜绝单点故障，扩展模块故障不影响核心；
2. **硬规则+幂等性**：从逻辑上确保执行结果可预测，避免重复/错误执行；
3. **自动回滚+熔断降级**：从故障处理上确保失败不扩散、状态可恢复；
4. **全链路监控+混沌测试**：从可观测性和验证上确保问题早发现、早解决。

### 关键保障要点
- 所有衔接逻辑**硬编码锁死**，无动态配置、无智能优化，仅做机械执行；
- 所有失败场景**提前定义兜底策略**，无“未知失败”，失败必回滚；
- 所有模块间仅通过**中文明文API**通信，无黑盒数据，问题可直接定位。

这套方案落地后，衔接流水线的可用性可达到99.999%，任意环节故障都能在1秒内触发兜底，用户仅感知“执行失败/降级”，但核心系统绝不崩溃、数据绝不脏、设备绝不乱。
